Time Manager UI Implementation Notes

Maintained during the create-ui-layer implementation.

Design Decisions

  • The UI is implemented as a local cockpit rather than a calendar-first dashboard. The first screen centers one current directive, the selected local agent, daemon state, cost, and evidence.
  • The web surface uses a Vite React frontend and a local Python API served by tm web. Web dependencies are optional so CLI-only installs remain light.
  • The implementation initially kept the dirty capture/privacy/dashboard branch as a contract reference; after the feature work was completed, those modules were imported and wired into real API/UI controls.
  • The completed capture/privacy/dashboard branch was imported into this UI branch after the initial UI implementation. The placeholders are now real API routes and visible UI controls.
  • Selected UI agent state is stored under the existing XDG data directory as web-config.json, not in tracked files or API-key environment variables.
  • The frontend design pass interprets the product as a private local command center, not a generic SaaS dashboard. The visual language now uses graphite surfaces, off-white data fields, trace-green/amber/red operational accents, compact mono labels, and a trace-spine motif tied to local event history.

Deviations

  • The dashboard panel includes a lightweight summary from existing repository APIs even before the capture branch is merged. The richer capture-branch dashboard remains feature-gated.
  • OpenClaw is shown as gateway/status context only. It is not a routeable LLM backend until the local gateway is healthy and a stable request contract is available.

Tradeoffs

  • Gemini and Kimchi adapters use prompt-enforced JSON extraction rather than native tool-use. This matches the existing Codex CLI limitation and keeps the shared LLMClient protocol stable.
  • The UI uses the daemon for LLM-backed write operations. If the daemon is down, the API returns a precise status instead of silently creating a second writer path.
  • Capture uploads send file contents as JSON and the backend writes temporary files so the UI can reuse the feature branch's tested CLI parsers. This avoids a multipart dependency and keeps command behavior single-sourced.
  • The frontend uses plain CSS rather than adding a component framework. That keeps the visual system explicit and avoids importing a generic SaaS look.
  • The design pass kept the existing React component boundaries instead of introducing a design-system dependency. This made the change mostly CSS plus small label updates, but it means future screens should reuse the same tokens and panel spans manually.
  • Running npm audit fix was intentionally skipped after install reported one low-severity advisory, because it can rewrite dependency choices outside the UI implementation scope.

Open Questions

  • Confirm whether OpenClaw should become a first-class scheduler/debrief backend after its local gateway is repaired, or remain an orchestration surface around other agents.
  • After the capture/privacy/dashboard branch is merged, confirm whether the UI should expose file upload for Telegram and calendar imports or only command-triggered imports from local paths.
  • Current answer implemented browser file upload for Telegram JSON and calendar ICS by reading the file in the browser and sending the content to local API endpoints.
