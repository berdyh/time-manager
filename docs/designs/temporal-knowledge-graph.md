# CEO Plan: Temporal Knowledge Graph for Personal Behavior

Generated by /plan-ceo-review on 2026-05-05
Branch: main | Mode: SELECTIVE EXPANSION
Repo: berdyh/time-manager

## Vision

A personal behavior model that applies **process mining** to a multi-signal personal event log to surface insights that text summarization can't reach: the actual control-flow of your days, conformance of intent vs reality, bottlenecks in your workflows, and predictions about today grounded in your own historical process model.

**This is NOT another LLM summarizer.** Direct LLM summarization (OpenClaw with `.md` files, ChatGPT with stored context, any chat tool with memory) already handles the "compress my day into prose" job. The differentiation here is **Personal and Human-Centric Process Mining (PHPM)** — a real subfield with its own academic conference workshop (1st edition at ICPM 2025) — applied to behavioral signals beyond text: timing, sequencing, recurrence, deviation, throughput, variant clustering. The LLM is the cheapest input source (it extracts events from natural-language debriefs), not the product. The product is what the process model reveals.

Two LLM agents (debrief at night, scheduler during the day) read and write a shared event log + discovered process model. The user talks; the model structures the conversation into events; process mining algorithms (Inductive Miner, conformance checking, predictive monitoring) discover what's actually happening. Insights surface inside the actionable moment ("today's trajectory matches your slow-Tuesday variant — you're likely to skip the workout again; want to move it earlier?") rather than in a separate dashboard.

Capture friction collapses because the user just talks. The event log is the system of record; the process model is the analytical artifact; the chat is the surface.

### 12-month dream state

```
Passive capture pipeline → small "behavior engine" → ONE high-confidence
hint per day, surfaced where the user already lives (Telegram bot or
calendar). Each hint accepted or ignored with one keystroke; that signal
feeds back into pattern confidence.
```

Not a dashboard. One well-placed nudge at the right moment, with a feedback loop that closes the same night. The "one hint per day" is mechanized as a hard rate-limit on **proactive** scheduler suggestions (see Engineering Details § Proactive Suggestion Rate-Limit). Reactive chat (user asks, system answers) is unconstrained.

## Engineering Details

### Process Mining Foundation (THE differentiation)

This section is the heart of the system. Read it first — every other engineering decision flows from here.

**Field grounding.** This is Personal and Human-Centric Process Mining (PHPM), an emerging subfield with a [first-edition workshop at ICPM 2025](https://icpmconference.org/2025/workshops/) and active research on wearable + calendar enrichment of personal event logs. Process mining itself is a 20+ year discipline (van der Aalst et al.) with established techniques and a mature open-source Python library, [PM4Py](https://processintelligence.solutions/static/api/2.7.17/getting_started.html) (`pip install pm4py`, paper: [Berti et al. 2023](https://www.sciencedirect.com/science/article/pii/S2665963823000933)).

**Event log structure** — XES-compatible so the data is portable to any PM tool (ProM, Disco, Celonis):

```
case_id          # which "case" this event belongs to (workday | goal-pursuit | activity-chain — see Case Notion below)
activity         # canonical activity name from a controlled vocabulary, e.g. "deep_work", "email", "exercise", "family_time"
timestamp        # ISO 8601, the temporal anchor
lifecycle        # start | complete | suspend | resume (PM4Py-standard)
resource         # location/device/who-present, where known
attributes       # arbitrary k/v from LLM extraction: planned_duration_min, actual_duration_min, mood, energy, blockers, etc.
extractor_version  # which prompt/schema produced this event (for re-extraction)
```

**Case notion (user-decided 2026-05-05): goal-pursuit + workday dual lens.** The "case" in process mining is the unit of analysis — what defines one trace. For personal behavior, v1 commits to **two concurrent case notions** because the user's life has two natural unit-of-analysis axes:

1. **Goal-pursuit case** — case = sequence of activities tagged with one goal_id, possibly spanning days or weeks. Discovers "what does it actually take to ship X?" — the project-workflow lens. Surfaces bottlenecks specific to this kind of goal, variants of how I usually pursue goals like this, time-to-completion predictions.
2. **Workday case** — case = one calendar day across all goals. Discovers "what does my typical efficient day look like?" — the daily-efficiency lens. Surfaces day-shape variants, day-level outcome clusters, daily bottlenecks (lunch→afternoon idle gap).

Implementation: each event has both `case:goal_id` (nullable, set when the user/LLM tags an activity to a goal) and `case:date` (always set from timestamp). PM4Py operates on either case-attribute as the case notion via `pm4py.format_dataframe(df, case_id='goal_id' | 'date')`. Same event log, two views, no duplication.

**Activity-chain case is deferred to v2** — the dual lens is enough to validate the hypothesis. Sliding-window analysis can be added later if the goal+workday lenses don't surface everything that matters.

**The four operations.** PM4Py provides primitives for all four; we wrap each in a `tm` command:

#### 1. Discovery — `tm discover`

Run **Inductive Miner** (`pm4py.discover_process_tree_inductive`) over the event log to produce a **Petri net** or **BPMN** model of the user's actual behavior. Inductive Miner is the modern default — guarantees soundness (no deadlocks), handles real-world noise, and is the recommended algorithm for messy logs (per [PM4Py docs](https://processintelligence.solutions/static/api/2.7.17/getting_started.html)).

Output stored in Kuzu as a graph:
- Nodes: activities, splits/joins (XOR/AND), start/end markers
- Edges: control-flow relationships with frequency counts and avg-duration weights

The discovered model is the user's **behavioral fingerprint**. Re-discover weekly (or on demand). Older models kept for trend comparison.

#### 2. Conformance Checking — replaces "match algorithm" entirely

For each new daily case (or any in-progress case), PM4Py runs **token-based replay** (`pm4py.conformance_diagnostics_token_based_replay`) or **alignment-based** check (`pm4py.conformance_diagnostics_alignments`) against the discovered model and produces:

- **Fitness** ∈ [0, 1] — how well this case fits the model. Low = unusual day.
- **Precision** ∈ [0, 1] — how restrictive the model is.
- **Named deviations** — "today you did `email` before `deep_work` (model expects opposite)", "today is missing `exercise` (model expects it 78% of weekdays)".

This replaces the cosine-similarity match algorithm with rigorous, established methodology. **The fitness score IS the implicit learning signal** — much harder to game or noise-poison than threshold matching.

#### 3. Variant Analysis — clusters days by process shape

`pm4py.get_variants` groups cases by their activity sequence. Real personal behavior has surprisingly few variants — humans are creatures of habit. Typical output: 5-12 distinct day shapes covering 80%+ of cases.

Then we cluster variants on **outcomes** (planned tasks completed, mood, energy):
- **"Good day shape"** = variants with high outcome scores
- **"Bad day shape"** = variants with low outcome scores

Insight surface: "Today's trace matches the slow-Tuesday variant (3 of last 5 with this shape ended without exercise). Want to break the pattern?" This is the actual causal-ish signal — clustered on real outcomes, not asserted patterns.

#### 4. Performance / Bottleneck Analysis — `tm bottlenecks`

`pm4py.discover_performance_dfg` annotates the discovered model with timing data:
- Throughput time per activity
- Waiting time between activities
- Idle gaps in the day

Surfaces actionable findings: "auth-bug-class tasks consistently run 2.8x your estimate", "the lunch→afternoon transition has a 45-min idle gap on 60% of weekdays" — concrete bottlenecks to address.

#### 5. Predictive Process Monitoring (LLM-based)

For real-time prediction during the day ("at 2pm, will I finish today's plan?"), use **LLM-PPM** following [Khan et al. 2026 — arXiv 2601.11468](https://arxiv.org/abs/2601.11468) which shows LLMs **outperform benchmark methods** on small-scale event logs (~100 traces) for total-time and activity-occurrence prediction. This is exactly our regime: at end of year 1 you have ~250 daily cases.

Implementation: serialize the partial day's prefix trace + a few similar historical traces into a prompt; ask the LLM to predict total time and remaining activity occurrences. LLM exploits both prior knowledge and trace correlations — no training needed. Cached daily.

#### 6. Prescriptive Process Monitoring (LLM-based, replaces ad-hoc scheduler)

Following the [Prescriptive Process Monitoring with LLM Explanations literature](https://link.springer.com/chapter/10.1007/978-3-031-70396-6_23) (PrPM): given current case state + discovered model + good/bad variant clusters, the LLM produces a **runtime intervention recommendation** with a natural-language explanation grounded in process facts ("move workout earlier — your slow-Tuesday variant skips it 80% of the time after 5pm").

The LLM is the explanation layer. The recommendation logic is grounded in PM (variant comparison, conformance deviations, bottleneck data). This is the "thinking with you, not at you" feel — but with a rigorous backbone.

### Why Kuzu Now Has a Real v1 Job

Earlier reviews flagged Kuzu as "architectural ballast" — graph DB without a clear use case. With process mining, Kuzu earns its keep:

- Stores the **discovered process model** as nodes (activities, splits, joins) and edges (control-flow with frequency + duration attributes). Cypher graph traversals power "what activities typically follow X?", "what's the longest path through my Tuesday variant?", "show me all variants where exercise occurred."
- Stores **variant graphs** — a meta-graph where nodes are variants and edges represent transitions between variants over time (you used to be in good-day-shape but trended toward bad-day-shape this month).
- Stores **process-discovery history** — last 12 weeks of weekly-rediscovered models so trends in fitness/precision can be plotted.

SQLite still owns event-log data (canonical, XES-compatible). Kuzu owns the **derived analytical artifacts** that are graph-shaped by nature.

### Outcome Model (user-decided 2026-05-05)

A KEY logical fix surfaced by the second Codex review (gpt-5.5): **conformance fitness ≠ "better day"**. If your discovered process model encodes bad habits (4 hours of scrolling at night), high fitness against that model means you had a *typical* day, not a *good* day. Using delta-fitness alone as the scheduler's success signal optimizes toward typical, not better.

The fix: every case has both a **conformance score** (sanity check that extraction worked + behavior is in distribution) AND an **independent outcome score**. Outcome is the real target. Conformance is the safety rail.

**Outcome score** — user's definition, simpler than ad-hoc weighting:

```
did_complete_tasks  = (planned_tasks_completed / planned_tasks_total) >= 0.80   # bool
did_advance_goal    = (any activity in this case has advances_goal != null)     # bool

outcome_score = int(did_complete_tasks) + int(did_advance_goal)
# 0 = bad day        (neither: didn't finish what you said you'd do AND didn't move any goal)
# 1 = good day       (one: either finished daily plan OR advanced a goal)
# 2 = great day      (both: shipped today AND moved the larger thing forward)
```

Why this is right and the 5-component formula was wrong: the user's framing is "shipping vs not, plus goal-progress vs not." Mood/energy/sleep are noise — they're inputs to a good day, not the definition of one. A great day with bad sleep is still a great day; the bad sleep is its own signal worth tracking but doesn't subtract from outcome.

**Variant clusters** (per case lens):
- Workday lens: cluster cases by activity sequence, compute mean outcome_score per cluster. "Good day shape" = mean ≥ 1.5. "Bad day shape" = mean < 0.5. "Mixed" = in between.
- Goal-pursuit lens: cluster pursuit cases by activity sequence, compute mean throughput-time and goal-completion rate. "Effective pursuit shape" = high completion + low throughput. "Stuck pursuit shape" = low completion + high throughput.

**Scheduler success metric**:
- Did the suggestion's day land in a higher-outcome variant cluster than the partial-trace was trending toward? Yes = success.
- Did the suggestion bump `did_complete_tasks` from 0 to 1 (the user finished what they planned)? Strong success.
- Did the suggestion bump `did_advance_goal` from 0 to 1 (the user moved a goal forward in a window where they otherwise wouldn't have)? Strong success.

Outcome delta is the primary signal. Conformance is the sanity check.

### Goals (v1 first-class)

The outcome model depends on goal tracking. Goals are first-class in v1 (previously planned for v2 backlog).

**Schema** — SQLite `goals` table:
```
goal_id TEXT PRIMARY KEY      # ulid or short slug
name TEXT NOT NULL            # "ship time-manager v1", "lose 5kg", "publish paper"
description TEXT              # optional richer text
status TEXT NOT NULL          # active | completed | abandoned
created_at TIMESTAMP
target_completion_at TIMESTAMP   # optional self-set deadline
completed_at TIMESTAMP        # set when status = completed
priority INT                  # 1-3, optional
```

**Activity tagging** — events get a nullable `advances_goal` attribute (foreign key to `goals.goal_id`). The LLM extracts this during debrief: when the user mentions an activity, the LLM matches against active goals; if uncertain, asks ("did 'wrote auth retry logic' advance the time-manager-v1 goal?").

**CLI**:
- `tm goal add "ship v1 of time-manager"` → creates active goal
- `tm goal list [--status active|completed|all]`
- `tm goal complete <id>`
- `tm goal abandon <id> --reason "..."`
- `tm goal show <id>` — pursuit history, throughput, current variant cluster, predicted completion (if goal has a target date)

**Process mining over goals** — each goal generates its own pursuit case in the event log. Discovery on the goal-pursuit lens shows "what does it take you to actually ship goals like this?" Conformance flags pursuits drifting from your typical effective pattern. Predictive monitoring estimates completion date based on the partial pursuit + similar past pursuits.

Effort: ~3 hr CC for schema + CLI + extraction prompt update. Goal hierarchy (sub-goals, dependency edges between goals) deferred to v2.

### Activity Vocabulary Governance (gpt-5.5 #5)

The activity vocabulary IS the product, not a schema detail. Too coarse and the discovered process is useless; too fine and variant counts explode and nothing recurs.

**Initial vocabulary** (10-15 activities, lockable post-init):
- `deep_work`, `shallow_work`, `meeting`, `email`, `code_review`
- `exercise`, `meal`, `commute`, `errand`
- `family_time`, `social`, `solo_recreation`, `learning`
- `rest`, `sleep`, `interruption`

**Governance loop** in v1:
- LLM extracts activities into the vocabulary with **soft alignment**: if the user's debrief mentions an activity not in vocabulary, LLM proposes the closest match + flags as "novel" with the user's exact wording stored in `attributes.original_label`.
- Weekly `tm vocab review` surfaces the top 10 most-frequent novel labels. User decides: merge into existing activity, create new activity, or ignore.
- Vocabulary changes get a version tag (`vocab_v1`, `vocab_v2`). Events store `vocab_version`. Re-extraction (v1.1) uses current vocab against historical transcripts.
- Vocabulary versioning is the same machinery as schema versioning — reuses the `tm reextract` foundation.

**Default safeguards**:
- No vocabulary auto-expansion. Adding a new canonical activity is always user-confirmed.
- Aliases table: user can map free-text variants ("workout"/"gym"/"run") to canonical (`exercise`).
- Drift detection: if a canonical activity stops appearing for 30+ days, the system asks "you stopped doing X. Archive or keep waiting?"

### Trace-Level Eval (gpt-5.5 #6)

Field-level extraction accuracy at 85% is **insufficient for process mining** because errors compound structurally — a single wrong activity label changes sequence, duration, variant assignment, conformance result, bottleneck calculation. The eval gate must measure trace-level validity, not just field-level.

**Eval set v2** (replaces the 50-debrief field-level gate):

| Metric | Target | What it measures |
|---|---|---|
| Field-level accuracy | ≥ 85% | Per-event property correctness (existing) |
| **Trace-level validity** | **≥ 70%** | Whole-trace match against hand-labeled gold trace (sequence + activity labels + key durations) |
| **Variant assignment accuracy** | **≥ 75%** | Does the extracted trace land in the same variant cluster as the gold trace? |
| Vocabulary novelty rate | < 10% per week (steady state) | High novelty = vocab not stable yet |

Eval set composition (from previous spec): ≥30 real + ≤20 synthetic. Each real debrief gets hand-labeled to the trace level (~5 min/debrief). One-time cost: ~3 hours of labeling.

If trace-level validity stays below 70% after prompt iteration, that's a **kill signal** for the LLM-as-extraction-layer hypothesis — not a bug to debug, a decision point: switch to passive sources or abandon the project.

### Kill Tripwires & Pivot Ladder (user-decided 2026-05-05)

A single kill threshold ("trace validity < 70% after 30 days = abandon") was too brittle and conflated two distinct things: tech failure vs project value. Replace with a portfolio of independent tripwires, each with a specific destination — most lead to a *pivot*, not abandonment. Only one (the value gate) leads to "shelve."

**Tripwires (each fires independently):**

1. **Value gate (day 30 hard-stop)** — by day 30 of consistent use, you must be able to name **≥3 unprompted "huh, didn't realize I do that" insights** AND **≥1 concrete behavior change traceable to a system suggestion**. Below either threshold → **shelve**, regardless of technical metrics. Trace fitness scores don't matter if the system isn't producing real-life value. This is the only outcome-based kill in the system. The other tripwires are pivots.

2. **Slope gate (during iteration)** — if trace-level validity is flat or declining for **2 consecutive weeks** while still below 70% → **pivot down the ladder** (next entry), not kill. The signal is the learning rate, not absolute level. A slow climber that's improving stays alive.

3. **Effort cap (defends against "one more prompt tweak")** — **10 hours total** of cumulative prompt iteration time. When the budget is spent, stop tweaking regardless of where the metric sits. Either you've converged, or you haven't, and more time isn't going to fix it. Tracked in `tm metrics` as `prompt_iteration_hours`.

4. **Adherence canary (calibration, NOT kill)** — if debrief completion is **<50% in the first 30 days**, the experiment can't conclude. Missed nights flag the *data* as inconclusive, not the *project* as failed. Response: simplify the debrief ritual (shorter prompts, fewer questions) or extend the window. The project keeps going.

**Pivot ladder (taken in order, each step is a serious change of approach, not abandonment):**

```
   free-prose debrief extraction
              │
        fails (slope gate fires)
              ▼
   structured-prompt debrief
   ("answer 5 specific questions" instead of "tell me about your day")
              │
        fails
              ▼
   passive-only sources
   (calendar import + screen-time API + browser history,
    no LLM debrief at all — structured signals only)
              │
        fails
              ▼
   shelve the project
```

The pivot ladder is encoded in code as a set of concrete config changes (extraction-mode flag in env), not a future redesign. v1 ships with `extraction_mode=free_prose` as default. Pivoting to `structured_prompt` is one config change + one swap of the debrief system prompt. Pivoting to `passive_only` requires the calendar/browser/screen-time adapters which are already roadmapped (v2 backlog) — pivoting accelerates them, not invents them.

**Why this structure beats a single threshold:**
- Tech failure ≠ value failure. Conflating them is how projects die when only one of the two failed.
- A flat-but-low metric is different from a declining metric. The slope gate distinguishes.
- The effort cap is a self-defense mechanism. Without it, "I'll just iterate one more week" runs to infinity.
- Adherence is a confounder, not a signal. Treating it as a kill metric punishes you for the wrong thing.

### Auth Model (Telegram)

- Bot reads `TM_ALLOWED_TELEGRAM_USER_IDS` from env (comma-separated list).
- All incoming messages from non-whitelisted IDs are silently dropped (no leak of bot existence). The dropped message content is NOT logged — only the rejected user ID and timestamp.
- API key for the LLM lives in `TM_LLM_API_KEY` env var; never logged; rotation via env update + daemon restart.
- Telegram bot token stored same way. If leaked, BotFather can re-issue; whitelist limits blast radius.
- **Daemon Unix socket**: file mode `0600`, owned by daemon user. `tm init` enforces this. Other users on the host cannot connect.
- **Prompt-injection hardening**: extraction prompts wrap user content in fenced delimiters (`<user_message>...</user_message>`) and the system prompt instructs the LLM to extract only structured fields per the schema, ignoring any instructions inside the user content. Output schema validation rejects fields outside the schema. This is best-effort, not foolproof — but it raises the bar.

### Privacy Trade-off: Telegram Cloud

Telegram bots are **not end-to-end encrypted**. Telegram's servers see plaintext of every message between you and the bot. This means:
- Your daily debrief content (what you did, why you couldn't focus, family routines, work details) flows through Telegram cloud before reaching your local daemon.
- Telegram does not commercially sell bot content, but they have it.
- For v1 this is an accepted trade-off in exchange for setup simplicity and zero frontend cost. Document this in README.
- v2 candidate: switch to Signal bot (E2EE) or local-only CLI for fully local capture. The LLM client abstraction means the rest of the system doesn't need to change.

### Extraction Failure Handling

```
LLM call ──► response received
              │
              ├─ valid structured output ──► commit to SQLite + Kuzu projection
              │
              ├─ malformed JSON ──► reprompt with parse error (max 2 retries)
              │                      └─ still failing ──► append to "stuck" queue,
              │                         user sees in next debrief: "I couldn't parse
              │                         your answer about X — try again?"
              │
              ├─ rate limit (429) ──► exponential backoff (1s, 4s, 16s), max 3 tries
              │                       └─ still failing ──► defer with user note
              │
              ├─ timeout ──► retry once, then defer
              │
              └─ refusal / safety filter ──► log + surface to user verbatim
```

No silent drops. Every failure either retries, defers with user-visible note, or surfaces the raw error. The "stuck" queue is itself a SQLite table.

**Stuck queue backpressure**: cap surfacing at **3 items per debrief** (most recent first). Items older than **30 days** in the stuck queue auto-archive to `stuck_archive` with a telemetry counter (`stuck_auto_archived_count`) so the metric is visible. User can browse archive via `tm stuck list --archived` if they want.

### Process Model Quality Metrics (replaces ad-hoc confidence math)

We do NOT invent confidence numbers. We use PM4Py's established quality dimensions of a discovered process model:

- **Fitness** ∈ [0, 1] — fraction of behavior in the log that the model can reproduce. PM4Py: `pm4py.fitness_token_based_replay(log, model)`.
- **Precision** ∈ [0, 1] — model doesn't allow much more behavior than was observed. PM4Py: `pm4py.precision_token_based_replay(log, model)`.
- **Generalization** ∈ [0, 1] — model handles unseen-but-plausible cases. PM4Py: `pm4py.generalization_tbr(log, model)`.
- **Simplicity** — Occam's razor for process models. Roughly 1 / number-of-arcs.

A discovered model is **promoted to "current"** when fitness ≥ 0.75 AND precision ≥ 0.6 over the last 30 days of cases (PM4Py defaults are well-studied; tunable per user via `tm config`). Below those, the model is "calibrating" and the scheduler operates in conservative mode (reactive only, no proactive nudges).

Re-discovery is weekly (Sunday night batch) plus on-demand via `tm rediscover`. Last 12 weekly models are kept for trend analysis.

**When new evidence breaks the model.** If a new daily case has fitness < 0.4 against the current model, that's a **drift signal** — the user's behavior is changing. Three weeks of low-fitness cases in a row triggers automatic re-discovery and a notification: "your Tuesdays look different now — re-derived your process model." This replaces the ad-hoc "+0.15 reinforcement / 14-day decay" — drift is detected mathematically, not by ad-hoc decay constants.

### Prescriptive Process Monitoring (replaces ad-hoc Pattern-Grounded Suggestion)

When the scheduler is asked "where should I fit task X?":

1. Compute the **current case state** — partial trace of today's events so far.
2. Query Kuzu for the **current discovered process model** + the variant clusters with their outcome scores.
3. Run **conformance check** of today's partial trace against the model — produces fitness + named deviations.
4. Run **LLM-based predictive monitoring** (per [arXiv 2601.11468](https://arxiv.org/abs/2601.11468) approach) — given the partial trace + similar historical traces (top-K by sequence similarity), predict total time and remaining activity occurrences for today.
5. **Variant lookup** — which variant cluster does today's partial trace most resemble? Is it a "good day shape" or "bad day shape"?
6. Construct the LLM prompt with: discovered model summary (top activities + most likely next activities + their throughput times) + matching variant cluster + conformance deviations + prediction. Ask for a recommendation with explanation.

The LLM weaves these into a single grounded suggestion: "Today looks like your slow-Tuesday variant (3 of last 5 ended without exercise). Predicted finish for the auth bug: 6:40pm — that's after your usual workout slot. Want to move workout to 4:30 or push the bug to tomorrow morning?"

This is **Prescriptive Process Monitoring with LLM explanations** — the LLM is the explanation/communication layer; the recommendation logic is grounded in process mining facts (variants, conformance, prediction). No arbitrary "top-3 patterns by confidence × overlap_score." The LLM decides what's actionable; the process mining decides what's true.

**Guardrails (gpt-5.5 #8 — preventing LLM-as-policy creep):**

The LLM is empowered to suggest, but not to invent. Three explicit guardrails:

1. **Objective function**: maximize predicted `outcome_score` (computed as defined in Outcome Model section). Every suggestion is scored: "if user accepts, predicted outcome bumps from N to M." Suggestions with negative or zero predicted-outcome delta are dropped before the LLM ever drafts a message.
2. **Counterfactual baseline**: for every suggestion, the LLM also receives the predicted outcome WITHOUT intervention (i.e., if the day continues on its current trajectory). Suggestions are only surfaced if predicted-with-intervention > predicted-without-intervention by a meaningful margin (default: ≥0.3 outcome-score points; tunable).
3. **Conformance-deviation guardrail**: a suggestion that pushes today into a trace shape with fitness < 0.4 against the discovered model is flagged "this would be unusual for you" — the LLM may still surface it, but with explicit acknowledgment ("this isn't your usual pattern, but..."). Prevents wild recommendations that look reasonable in prose but are out-of-distribution for the user.

Telemetry per suggestion records: predicted_outcome_with, predicted_outcome_without, predicted_outcome_delta, conformance_deviation, accepted_thumbs, actual_outcome_at_eod. Over time, the system learns which guardrail thresholds align with actual user satisfaction and tightens or loosens them.

### Suggestion Outcome Signal (replaces ad-hoc Implicit Accept)

When the scheduler proposes "do X around 2pm", the outcome signal comes from process mining, not cosine matching:

- **Conformance fitness on today's case after the suggestion**: did the suggestion improve today's fitness against the current model? PM4Py replay produces a fitness number for the partial trace before the suggestion was issued and another after the day completes. Delta-fitness is the signal.
- **Variant trajectory**: did the day's final variant cluster shift from a bad-day-shape variant (where it was trending pre-suggestion) to a good-day-shape variant?
- **Activity occurrence prediction**: was the LLM-PPM prediction (e.g., "you'll finish auth bug by 5pm") correct? Stored alongside the suggestion; reconciled at end-of-day.
- **Explicit thumbs** = ✅/❌ button on the bot suggestion. Strong signal, overrides implicit.

Telemetry row:
```
(suggestion_id, suggested_at, partial_trace_at_suggestion, recommended_action,
 day_final_variant, day_final_fitness, predicted_completion, actual_completion,
 explicit_thumbs, llm_explanation_text)
```

This is far more rigorous than cosine-similarity matching:
- No threshold guessing. Conformance fitness is a continuous, calibrated number.
- No "did the user do exactly the thing I suggested" tax. The signal is "did the day go better?", not "did you literally do X at 2pm".
- Reconciles naturally with PM literature on **prescriptive process monitoring evaluation** — measure intervention impact via conformance + outcome metrics.

### Proactive Suggestion Rate-Limit

The scheduler agent has two modes:
- **Reactive** (user asked something): unconstrained. Respond as helpfully as possible.
- **Proactive** (system-initiated nudge): hard cap of 1 per day, gate on `MAX(pattern.confidence) ≥ 0.8`. If no hardened pattern is relevant, no proactive nudge fires that day.

This mechanizes the "ONE high-confidence hint per day" vision.

### Store Facade Interface

One interface, two backends underneath. Approximate method list:

```python
class Store:
    # Events (writes go to SQLite; Kuzu projection updated async)
    def append_event(event: Event) -> EventId
    def get_event(id: EventId) -> Event | None
    def query_events(filter: EventFilter) -> list[Event]

    # Conversations (SQLite only)
    def append_conversation(turns: list[Turn], metadata: ConvMeta) -> ConvId
    def get_conversation(id: ConvId) -> Conversation | None

    # Patterns — ALL READS FROM SQLITE (canonical). Kuzu pattern mirror exists for
    # graph traversal queries only, never for pattern lookup.
    def upsert_pattern(pattern: Pattern) -> PatternId
    def reinforce_pattern(id: PatternId) -> None
    def query_patterns(context: Context, min_confidence: float, limit: int) -> list[Pattern]

    # Graph queries (Kuzu only; falls back to SQLite recursive CTE if Kuzu disabled)
    # Used for multi-hop relationship walks, NOT for pattern reads.
    def graph_walk(start: NodeId, edge_types: list[str], max_hops: int) -> list[Node]

    # Telemetry (SQLite only)
    def log_suggestion(suggestion: Suggestion) -> SuggestionId
    def record_signal(suggestion_id: SuggestionId, signal: Signal) -> None

    # Maintenance
    def rebuild_kuzu_projection() -> None  # idempotent
    def export(format: Literal["jsonl", "cypher"]) -> Path
    def forget_event(id: EventId) -> None  # cascades to Kuzu
    def redact_range(start: datetime, end: datetime) -> int  # returns count redacted
```

SQLite is the only store the daemon writes to directly. Kuzu is rebuilt from SQLite events nightly (and on demand). This solves the dual-write consistency problem by making Kuzu derived, not authoritative.

### Resilience

Failure paths the daemon must handle gracefully (anything else can crash with a stack trace, but these get explicit handling):

- **In-flight LLM call across daemon restart (1A)**: before each LLM call, daemon writes `(call_id, conversation_id, prompt_hash, started_at)` to an `inflight_calls` SQLite table. After response received and committed, the row is removed. On daemon startup, any rows older than 5 minutes are surfaced to the user as "extraction interrupted — retry?" Rows newer than 5 min are assumed possibly-still-in-flight and the daemon waits up to the LLM provider timeout before treating them as failed. No silent loss.
- **Disk full (1B)**: before each SQLite write, daemon checks free space against a configured floor (default 100 MB). If below floor, write is refused with a user-visible "storage full — free disk space and retry" message via the bot. Backups also check before writing.
- **Keyring unavailable on startup (1C)**: if `keyring.get_password(...)` raises (no D-Bus session, headless boot, etc.), daemon refuses to start and prints clear remediation: "Could not read SQLCipher key from system keyring. Ensure D-Bus / login session is active, or run `tm init --rekey` to recover from passphrase." Daemon does NOT silently fall back to unencrypted DB.
- **Kuzu projection integrity on startup (1A-eng)**: daemon startup compares `SELECT MAX(event_id) FROM events` (SQLite) against `MATCH (e:Event) RETURN max(e.id)` (Kuzu). On mismatch, log both values and rebuild Kuzu via `rebuild_kuzu_projection()` before serving requests. Catches all "daemon crashed mid-Kuzu-write" scenarios that would otherwise produce a stale projection. Rebuild is O(events); flag if it exceeds 30s wall time and switch to incremental update at that point.

### Semantic Extraction Confirmation (1B-eng)

Field-level eval (≥85% on real debriefs) catches schema correctness but not semantic correctness — the LLM might extract "Saturday family time" when the user said "Saturday family lunch." Both pass schema validation; the wrong one shapes patterns.

- Each evening debrief opens with: **"Yesterday I extracted: lunch with mom Saturday 1-3pm; reviewed PR #42 morning. Anything wrong?"**
- One-line yes/no/edit response. Yes / silence-after-60s = confirmed. No → user states correction inline; LLM updates the corresponding event, audit row written. Edits do NOT reinforce the originally-extracted pattern (the rate-limit slot for that pattern stays open).
- Catches semantic drift within 24 hours instead of weeks-after-pattern-hardens.

### Code Discipline (Section 2 eng findings)

Cultural rules baked into the project from day 1:

- **Reader/Writer separation at the type level (2A-eng)**: `StoreReader` is a `typing.Protocol` with only read methods (`get_event`, `query_events`, `query_patterns`, `graph_walk`, etc.). `StoreWriter` is the concrete class with read + write methods. Bot and CLI are typed as `StoreReader` so static type-checking (`mypy`/`pyright`) catches accidental writer calls. Daemon is the only consumer of `StoreWriter`.
- **Versioned Pydantic schema layout (2B-eng)**: schemas live in `tm/models/v1/`, `tm/models/v2/`, etc. Every extraction record stores `schema_version` (e.g. `"v1"`). Extractor reads the version-pinned schema. Schema migration = write `models/v2/` + ship `tm reextract --to-version v2` (v1.1).
- **Lint discipline (2C-eng)**: `ruff` config forbids `except Exception`, `except BaseException`, and `from x import *`. Pre-commit hook runs `ruff check`. Forces named-exception discipline from day 1.

### Test Discipline (Section 3 eng findings)

- **Test stack**: `pytest` + `pytest-asyncio` + `hypothesis`. `tests/properties/` directory holds property-based tests.
- **Property tests for math (3A-eng)**:
  - `confidence.py`: bounds `[0, 1]` always; reinforcement is monotonic non-decreasing within rate-limit window; decay over N days produces expected EWMA value within tolerance; threshold transitions (0.4 / 0.8 / 0.2) fire at exactly the right values.
  - `match.py`: cosine score symmetry; threshold-band activations fire correctly; LLM-fallback path triggers only in `[0.65, 0.78)`.
- **Eval-set companion repo (3B-eng)**: `time-manager-evals` private repo, encrypted at rest, holds the 50-debrief eval set (≥30 real + ≤20 synthetic). CI in the main repo pulls the evals repo, runs the extractor against the eval set, gates ship at ≥85% real-debrief field accuracy. Regression check runs on every extractor version bump.

### Performance Note (Section 4)

Personal scale, no real performance concerns. One operational watch: if `rebuild_kuzu_projection()` ever exceeds 30s wall time (currently O(events), <5s at v1 scale), surface it in `tm metrics` and consider switching to incremental Kuzu update. Not v1 work; v2+ if it ever happens.

### Concurrency

Single-writer daemon process. SQLite uses WAL mode. **Events writes use short transactions** (no `BEGIN IMMEDIATE` for individual event appends — let WAL handle MVCC); only **batch jobs** wrap their work in `BEGIN IMMEDIATE` so they hold the write lock for the duration of the batch and bot/CLI clients see consistent state.

- **Writes**: bot and CLI send commands to the daemon over a Unix domain socket. Daemon is the only process that opens SQLite for write.
- **Reads**: bot and CLI open SQLite directly in WAL read mode (multi-reader concurrent with the daemon's writer). Reads do NOT round-trip the socket — that would add latency for every render.
- **Store facade**: the `class Store` interface lives in a shared library used by daemon (write methods) and clients (read methods). Write methods on the client side proxy to the daemon over the socket; read methods open SQLite directly.

This eliminates dual-process write conflicts and "did the bot or CLI write this first" ambiguity, while keeping reads fast.

### Cost Ceiling

- Hard cap: `$20/month`. Tracked daily in SQLite via `(date, model, input_tokens, output_tokens, est_cost_usd)`.
- **Pre-call gate**: before each LLM call, daemon estimates max cost from `len(prompt_tokens) * input_price + max_output_tokens * output_price`. If `month_total + estimate > cap`, refuse without making the call. After call returns, reconcile estimate with actual.
- **Soft alarm** at 70% of monthly cap → daemon sends Telegram message: "you're at $14/$20 this month."
- **At 100%** → daemon refuses LLM calls for the rest of the month, surfaces "cost cap hit" to user.
- **Override**: `tm cost --override-month 2026-05 --new-cap 30` raises the cap for the named month. Every override writes a row to `cost_overrides (month, old_cap, new_cap, set_at, reason)` for audit. Override applies only to the named month — does not propagate.

### LLM Client Abstraction

Three operations, narrow internal contract — no pretending wire formats are universal:

```python
class LLMClient(Protocol):
    def chat(messages: list[Message], system: str) -> str
    def extract(messages: list[Message], schema: dict) -> dict  # provider-specific tool-use
    def tool_call(messages: list[Message], tools: list[Tool]) -> ToolCall
```

Two adapter families ship in v1:

**Direct-API adapters** (default, lowest latency):
- `ClaudeAdapter` — Anthropic Messages API + tool-use for `extract()`
- `OpenAIAdapter` — OpenAI Chat Completions + function-call for `extract()`

**ACP / agent-harness adapters** (opt-in via `TM_LLM_BACKEND` env var):
- `ClaudeCodeAdapter` — spawns `claude -p "<prompt>"` subprocess, parses output
- `CodexAdapter` — spawns `codex exec "<prompt>"` subprocess, parses output

**Deferred to v1.1**:
- `OllamaAdapter` (local LLM via HTTP)
- `OpenCodeAdapter` (opencode subprocess)

**Backend selection**: `TM_LLM_BACKEND` env var (`anthropic` | `openai` | `claude-code` | `codex`). Default: `anthropic`. The daemon binds one backend at startup; switching requires daemon restart.

**Why direct-API is default**:
- Subprocess spawn adds 100-500ms latency per call, which hurts scheduler chat feel.
- Token counts come for free in API response headers; CLI stdout requires parsing harness-specific format.
- Streaming (for long replies) is cleaner over HTTP.

**Why ACP adapters are first-class anyway**:
- Reuses the user's existing agent harness auth (no separate Anthropic/OpenAI key needed in `tm`'s env).
- Privacy posture: prompts go through the local harness (which the user already trusts), not directly from `tm` to provider.
- Lets the harness add MCP servers, tool use, multi-step reasoning if richer extraction is wanted later.
- Trivial model swap: change harness config, `tm` keeps running.

### ACP Adapter Implementation Notes

For each ACP adapter the implementation handles:

- **Subprocess hygiene**: spawn with explicit working directory, no inherited env beyond a controlled allowlist, hard timeout (default 60s for chat, 120s for extract), SIGKILL on timeout.
- **Output parsing**: each harness has its own stdout format. Adapter knows the harness's "final answer" delimiter or JSON envelope. If parsing fails, surface as `ParseError` (same path as malformed direct-API output → reprompt → stuck queue).
- **Tool-use mapping**: for `extract()`, the adapter constructs a prompt that embeds the schema (e.g., as a JSON-Schema block in the system instruction) and asks the harness to return a JSON object only. If the harness supports tool/function calling internally, the adapter can pass the schema as a tool definition; otherwise it relies on the harness's general instruction-following.
- **Cost visibility**: best-effort. Each adapter parses any token-usage line the harness emits (e.g., claude code's "tokens used: in/out" footer). If the harness doesn't expose token counts, fall back to a rough estimate from prompt length. Cost-cap pre-check uses the estimate.
- **Streaming**: not used for `extract()` (we wait for the full structured response). For `chat()` (scheduler reactive replies), adapter streams stdout if the harness streams; otherwise falls back to whole-response delivery.
- **Failure mapping**: subprocess exit code != 0 → `LLMBackendError` with stderr captured. Same retry/backoff/stuck-queue semantics as direct-API failures.

The adapter file layout: `tm/llm/claude_adapter.py`, `tm/llm/openai_adapter.py`, `tm/llm/claude_code_adapter.py`, `tm/llm/codex_adapter.py`. Each adapter is independently testable with subprocess mocking (or fakes that emit canned harness output).

### Encryption (at rest)

- **SQLCipher** for SQLite (transparent AES-256). Kuzu is rebuildable from SQLite, so Kuzu's database files don't need separate encryption; they hold only derived data.
- **Key storage**: OS keyring (libsecret on Linux, Keychain on macOS) via `keyring` Python lib. Daemon reads the key on startup, holds it in memory while running.
- **Key bootstrap**: first run of the daemon prompts the user to set a passphrase via the CLI (`tm init`). Passphrase is hashed (Argon2id) into the SQLCipher key and stored in the OS keyring. Passphrase itself is never stored.
- **Crash recovery**: SQLCipher uses WAL mode; if the daemon crashes, the next start re-opens the encrypted DB with the keyring-stored key. No re-prompt unless the keyring entry is lost.
- **Backup encryption**: backups are produced from the already-encrypted SQLCipher file plus the Kuzu directory. The backup copy is then re-encrypted with `age` using a separate user-supplied recipient key — keyring-stored daemon key is NOT exported with backups (defense in depth).

### Schema Migrations

- **SQLite**: numbered migration files (`migrations/001_initial.sql`, `002_add_pattern_decay.sql`...). Daemon runs unapplied migrations on startup. Daemon takes an exclusive `BEGIN IMMEDIATE` lock during migration; concurrent CLI/bot read attempts retry on `SQLITE_BUSY` with backoff (≤3 retries, then surface "daemon migrating, try again in a moment"). For a single-host personal product this is sufficient.
- **Kuzu**: no migration tool. On schema change, drop and rebuild the Kuzu projection from SQLite (via `rebuild_kuzu_projection()`). Acceptable because Kuzu is derived. Rebuild runs as part of the same daemon-startup migration sequence; the daemon does not start serving until Kuzu rebuild is complete.

### Nightly Batch Trigger

- Linux: systemd timer (`tm-nightly.timer` → `tm-nightly.service`). Service calls `tm batch --nightly`.
- macOS: launchd plist (deferred — daemon also exposes `tm batch --catchup` to run any missed batches at next daemon start).
- If the host was asleep at the scheduled time, the daemon detects missed runs on next start and catches up.

### Extraction Quality Eval

Before declaring v1 "shipped":
- Hand-label 50 debriefs total: **≥30 real** (collected over 1-2 weeks of using the system in shadow mode where extraction runs but doesn't update patterns) + **≤20 synthetic** (for adversarial edge cases: terse user, venting, contradictory statements, mixed-language).
- Report accuracy split by source: real vs synthetic. Synthetic accuracy can be higher and that's fine; the real number is what gates shipping.
- **Target: ≥85% field accuracy on real debriefs.** Below that → tune prompts and/or schema before shipping the loop to actual daily use.
- Re-evaluate when the extractor is bumped to a new version. Old eval set is preserved for regression testing.

### Voice, First-Run, and Empty States

**Bot voice spec** (baked into both DebriefAgent and SchedulerAgent system prompts):
- Short, direct, no decorative emoji. The only emoji are the ✅/❌ on suggestion buttons.
- No filler ("Got it!", "I've added that to your knowledge graph!", "Great question!"). The system acknowledges by acting, not by announcing.
- One thought per message. Never a wall of text.
- Match user's tone: if they're terse, be terse. If they're venting, listen don't probe.
- Reasoning second, action first. "Slot it for 9am — you finish hard tasks before 11 most days" not "Based on your patterns I've calculated that mornings are optimal..."

**First-run guided session** (triggered on first `tm init`):
- 6 framing questions answered in the bot, used to seed initial preferences AND goals:
  1. "What hours do you typically work?" (start, end)
  2. "What does a 'good day' look like for you?" (free text, stored as initial goal context)
  3. **"Name 1-3 goals you're actively pursuing right now."** (e.g., "ship time-manager v1", "lose 5kg", "finish PhD chapter 3"). Each becomes an active row in `goals` table. Activity→goal tagging works from Day 1 instead of bootstrap-empty. **Goals can be added, edited, or completed any time later via `tm goal add/edit/complete` — first-run is a starting point, not a hard limit.**
  4. "Any recurring commitments I should know about? (e.g. Saturday family time, Wednesday dinner)" (seed patterns)
  5. "When should I check in with you for the daily debrief?" (sets a default time for proactive suggestion gating)
  6. "Pick your initial activity vocabulary scope" (default: 16-activity starter set; advanced users can edit)
- Answers stored in SQLite `user_prefs` table. Patterns from question 4 marked `source = 'user_stated'` and start at confidence 0.6 (above tracked threshold but below hardened — they don't proactively surface yet but inform context overlap). Goals from question 3 are first-class rows; nothing special about init-time vs later-added.

**Scheduler empty-state** (no hardened patterns yet, ~first 2 weeks):
- Proactive scheduling is silent (no daily nudge fires) until at least one pattern reaches confidence ≥ 0.8.
- Reactive scheduler ("where do I fit X?") works but flags uncertainty: "I'm still learning your patterns — this is a guess based on N data points so far."
- During this period the debrief ends with a small progress nudge: "Day N of pattern-learning. After about 2 weeks of consistent debriefs the scheduler will start surfacing things proactively."

### Deletion + Redaction

- `tm forget --event <id>`: removes event from SQLite events table, **surgically removes** the corresponding node from Kuzu via `MATCH (e:Event {id: $id}) DETACH DELETE e`. Tombstone row written to `event_tombstones (event_id, deleted_at, reason)`. Full Kuzu rebuild only happens on schema change, not on every forget.
- `tm redact --range <from> <to>`: zeros out transcript text in conversations table for the time range; structured events extracted from those transcripts remain by default. With `--cascade`, events extracted from redacted conversations are also removed (each via the surgical Kuzu delete above).
- `tm forget --pattern <id>`: marks pattern archived AND writes an entry to `forgotten_patterns (signature, reason)` that the extractor reads as a system-prompt addendum: "user has explicitly forgotten patterns matching: ... — do not re-extract these."

## NOT in scope (rejected from v1)

- **Neo4j as primary store** — Overkill at personal scale (<500k events/year). Kuzu gives the same Cypher DX without JVM/ops overhead.
- **Formal causal inference framework** (DoWhy, EconML) — N=1 data with no possibility of randomization makes formal causal inference largely aspirational. Patterns + accept/reject telemetry give 90% of the practical value with 5% of the complexity. Re-evaluate at v3 if a clear causal hypothesis demands it.
- **Web dashboard** — The chat IS the surface.
- **Local LLM on day 1** — Quality gap at <30B params hurts extraction. Wrap the client behind an interface so swap is possible later.
- **Multi-user / shareable** — Personal product. Multi-user adds auth, isolation, schema migrations, ~5x effort with no clear demand.
- **`tm reextract` tool itself** — Foundation (versioned extractor + transcripts kept) is in v1. The actual re-extraction CLI command is v1.1 — write it when the first extractor version is superseded.

| Encryption at rest | **SQLCipher chosen** — transparent encrypted SQLite (see Engineering Details § Encryption) via `pysqlcipher3`. Key in OS keyring via `keyring` stdlib. |
| Time-series queries | SQLite recursive CTEs (DuckDB if speed pressure later) |
| Migrations | sqlite-migrate or yoyo-migrations (light, single-file tools) |
| Daemon scaffolding | systemd unit + Unix socket — no framework needed |
