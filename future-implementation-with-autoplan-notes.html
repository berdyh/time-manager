Decisions, Deviations, Tradeoffs, Open Questions

Type Note Impact
Deviation $autoplan was treated as blocked, not simulated. Prior project memory and the skill file both require a real AskUserQuestion tool for premise and final approval gates. The branch has code/tests plus this audit file, but not a formal gstack autoplan review artifact.
Design decision Capture is local-first: tm capture telegram imports Telegram JSON exports, tm capture calendar imports local .ics files, and tm capture voice stores already-transcribed voice notes. Ships useful ingestion without adding a bot, CalDAV OAuth, or audio transcription service in this pass.
Design decision Telegram and calendar capture imports use deterministic event IDs from source metadata, so rerunning the same export skips duplicates. Telegram IDs are scoped by chat/export identity; exports without a top-level chat ID/name use stable non-content message metadata hashes rather than a local file path, whole-export hash, or message text hash. Re-importing local files is safe for dashboard and variant counts without adding a separate import ledger or leaking workstation paths into exported data.
Design decision Voice capture uses random event IDs unless an explicit source ID is available; raw transcript text is not embedded in deterministic IDs. Avoids leaking captured text through identifiers while preserving deterministic imports for sources that provide stable metadata.
Tradeoff SQLCipher/keyring support is optional and fail-closed. A configured key is applied on project-owned SQLite connections, including backup and reextract staging destinations, but the command reports an error if the local driver does not expose SQLCipher. tm encryption set-key prompts for the key and does not accept it on the command line. Keyring storage is completed before a new encrypted database file is created. Project-opened SQLite database files and existing sidecars are hardened to owner-only permissions. SQLite paths are expanded and resolved once, then the same path is used for key lookup, file creation, opening, and permission hardening; the set-key command uses that normalized path before refusing existing databases or writing keyring entries. The temporary umask used around SQLite creation/WAL setup is process-wide, so it is guarded by a global lock for daemon worker threads. Avoids a heavy dependency while preventing a false sense of encryption on normal stdlib SQLite builds, and avoids leaving an unusable encrypted DB when keyring storage fails. The default tm data directory is also owner-only.
Design decision tm reextract deletes only prior debrief-derived events for a case date after a retained transcript has replayed successfully in a staged database. If final production replacement fails after the staged LLM call, staged cost rows are still appended to the real cost ledger. Any privacy action committed during staging aborts the final replacement under a write lock, so privacy wins over stale reextract snapshots. Calendar, Telegram, and voice capture events for the same day are preserved unless the user explicitly runs privacy deletion, and a failed replay leaves the previous debrief rows intact.
Design decision Debriefs run through either the CLI or daemon retain the source transcript only after extraction succeeds. Keeps tm reextract usable for daemon-created cases without letting failed or duplicate debrief attempts overwrite the retained transcript.
Tradeoff Retained transcripts are still keyed by case_date, but same-day writes now merge text instead of replacing it. Duplicate detection compares exact retained transcript segments, not arbitrary substrings, upserts take an immediate SQLite write transaction, and transcript write paths use the shared SQLite sidecar hardening path. This preserves replay material for a day with both debrief and voice capture, including short notes that appear inside longer debrief text, while avoiding a broader transcript-table redesign in this branch.
Design decision Dashboard metrics include retained transcript counts, filtered by the same --since/--until case-date window used for events and suggestions. Keeps the dashboard aligned with transcript retention, privacy, and reextract behavior without adding a larger reporting surface.
Design decision Privacy redaction and forget clear the local Kuzu projection path after staging SQLite mutations but before committing them. Kuzu is treated as rebuildable derived storage, so clearing the projection is simpler and safer than trying to surgically edit graph artifacts per redacted event.
Design decision Custom Kuzu projection cleanup is allowed only for paths marked with .tm-kuzu-projection. File-backed Kuzu paths use a sibling marker; directory-backed paths use an internal marker. Symlinked projection paths are refused before any privacy mutation. Kuzu cleanup runs before SQLite commit, before best-effort VACUUM/wal_checkpoint purging. Privacy --case-date selectors use strict YYYY-MM-DD validation instead of no-oping on malformed dates, and they also match legacy events whose case_date column still has the empty-string sentinel by using the event timestamp's half-open day window. Privacy mutations acquire BEGIN IMMEDIATE before selecting affected rows, so concurrent captures cannot slip between selection and mutation. For --event-id, legacy rows with an empty case_date derive the associated transcript/suggestion day from the event timestamp. Busy WAL checkpoint results surface as purge warnings. forget --event-id reports transcript deletion when it derives and clears the event's case transcript. Keeps privacy commands from deleting arbitrary user-supplied paths while still clearing project-owned derived storage even if SQLite post-delete purging is temporarily blocked by another process. If Kuzu cleanup itself fails, the SQLite privacy transaction rolls back.
Tradeoff Calendar capture rejects TZID-qualified and recurring events, plus floating datetimes, instead of guessing timezone conversion or recurrence expansion. Date-only calendar values are parsed before persistence so impossible dates are rejected. Telegram capture prefers date_unixtime and does not treat naive export date strings as UTC. This avoids corrupting timestamps and case dates. Full timezone conversion should be added before importing local-time calendar exports at scale.
Tradeoff tm variants --trend requires explicit --since and --until, then compares with the previous equal-size window. Keeps trend labels deterministic and avoids guessing the user's intended comparison period.
Design decision Export and backup outputs reject the source database path and SQLite sidecars (-wal, -shm, and -journal) after canonicalizing the source database path. JSON export reads all exported tables through one connection and one read transaction. Prevents JSON export or backup overwrite from clobbering live SQLite state while the database is in WAL mode.
Design decision Daemon detaching is POSIX double-fork behind the existing --no-foreground flag. Foreground mode remains the default for systemd, launchd, and tests. Start, stop, and status all expand and resolve pid/socket paths consistently. Explicit daemon paths do not force resolution or creation of the default data directory. Adds proper self-detach without disrupting supervised deployments.
Deviation Live-LLM eval thresholds are separated from mock thresholds in the eval harness, but no fresh live run was possible because TM_LLM_API_KEY is not set in this environment. The branch ships calibration machinery and conservative live gates; the real eval companion repo should refresh those numbers.
Open question Should Telegram become a long-running bot/webhook, or is local export import enough for the first daily workflow? Bot auth, allowlists, and message correction UX are intentionally left out until that product choice is clear.
Open question Should voice capture own transcription, or should it continue to accept text from the user's preferred phone/OS transcription path? Owning transcription adds model/runtime/privacy decisions that are larger than this branch.
